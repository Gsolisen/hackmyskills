       name: Validate Content

             - name: Run content tests

    2. Create `docs/content-generation-guide.md` with the following structure and content:

       Title: "# Content Generation Guide for HackMySkills"

       Section "## Overview":
       - Questions are generated by the maintainer using Claude Code sessions
       - Generated questions are written directly into existing YAML content files
       - Review happens in git diff before committing
       - No end-user-facing AI dependency -- all content ships as bundled YAML

       Section "## Question Schema":
       - Required base fields: id, type, topic, tier, tags, version_tag, last_verified
       - Optional field: source (values: "ai" or "curated"; omit for curated)
       - AI-generated questions MUST include: `source: ai`
       - ID format: `{topic}-{subtopic}-{type_hint}-{number}` (e.g., `k8s-rbac-scenario-003`)
       - version_tag: technology version the question targets (e.g., `k8s-1.29`, `tf-1.7`)
       - last_verified: date the answer was last confirmed accurate (ISO format: "YYYY-MM-DD")

       Section "## Question Types" with four subsections, each with:
       - Required fields
       - Example YAML block
       - Quality guidelines

       ### Flashcard (type: flashcard):
       - Fields: front, back
       - Front: clear question, single concept
       - Back: concise answer (1-3 sentences)
       - Example:
         ```yaml
         - id: k8s-service-types-001
           type: flashcard
           topic: kubernetes
           tier: L1
           tags: [services, networking]
           version_tag: k8s-1.29
           last_verified: "2026-01-01"
           source: ai
           front: "What are the four Service types in Kubernetes?"
           back: "ClusterIP (default), NodePort, LoadBalancer, ExternalName"
         ```

       ### Command Fill (type: command-fill):
       - Fields: prompt, command, accept_partial
       - prompt: describes what the user needs to accomplish
       - command: the exact command expected
       - accept_partial: true if partial match is acceptable
       - Example:
         ```yaml
         - id: tf-state-list-001
           type: command-fill
           topic: terraform
           tier: L1
           tags: [state, cli]
           version_tag: tf-1.7
           last_verified: "2026-01-01"
           source: ai
           prompt: "List all resources currently tracked in the Terraform state"
           command: "terraform state list"
           accept_partial: true
         ```

       ### Scenario (type: scenario):
       - Fields: situation, question, answer, explanation
       - situation: sets the scene (real-world context)
       - question: what to do or diagnose
       - answer: the correct response
       - explanation: why this is correct
       - Example:
         ```yaml
         - id: k8s-pdb-scenario-001
           type: scenario
           topic: kubernetes
           tier: L2
           tags: [availability, pdb]
           version_tag: k8s-1.29
           last_verified: "2026-01-01"
           source: ai
           situation: "You need to deploy a critical update, but must maintain at least 2 replicas running at all times during rollout."
           question: "What Kubernetes resource should you create?"
           answer: "PodDisruptionBudget with minAvailable: 2"
           explanation: "A PodDisruptionBudget (PDB) tells the cluster how many pods must remain available during voluntary disruptions like rolling updates or node drains."
         ```

       ### Explain Concept (type: explain-concept):
       - Fields: concept, model_answer
       - concept: the term or idea to explain
       - model_answer: a thorough explanation (3-5 sentences)
       - Example:
         ```yaml
         - id: tf-state-concept-001
           type: explain-concept
           topic: terraform
           tier: L2
           tags: [state, fundamentals]
           version_tag: tf-1.7
           last_verified: "2026-01-01"
           source: ai
           concept: "Terraform state locking"
           model_answer: "State locking prevents concurrent operations on the same state file. When using a remote backend like S3 with DynamoDB, Terraform acquires a lock before any write operation. This prevents race conditions where two applies could corrupt the state. If a lock cannot be acquired, Terraform will retry or fail depending on configuration."
         ```

       Section "## Difficulty Tiers":
       - L1 (Recall): Direct knowledge recall. "What is X?", "What command does Y?". Tests terminology and basic commands. Suitable for beginners.
       - L2 (Application): Apply knowledge to a situation. "Given X scenario, what would you do?". Requires understanding of how components interact. Suitable for practitioners.
       - L3 (Scenario): Complex multi-step reasoning. Real-world debugging, architecture decisions, trade-off analysis. Requires deep understanding and experience. Suitable for senior practitioners.

       Section "## Generation Workflow":
       1. Choose a topic and tier
       2. Open a Claude Code session
       3. Ask Claude Code to read this guide and existing YAML examples
       4. Request N questions of the specified tier
       5. Claude Code writes them directly into the appropriate YAML file
       6. Review the git diff for quality, accuracy, and formatting
       7. Run `hms validate-content` to check schema and duplicates
       8. Commit if valid

       Section "## Quality Checklist":
       - [ ] ID follows `{topic}-{subtopic}-{type_hint}-{number}` pattern
       - [ ] Tier matches difficulty guidelines above
       - [ ] version_tag reflects the technology version tested
       - [ ] last_verified date is set to today
       - [ ] source: ai is present
       - [ ] Answer is technically accurate and current
       - [ ] No duplicate with existing questions (run validate-content)
       - [ ] Tags are relevant and use existing tag conventions

       Section "## Topics":
       List current topics: kubernetes, terraform (with note that CI/CD, bash, AWS are planned for Phase 6)
